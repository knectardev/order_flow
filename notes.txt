# Cursor Prompt: Migrate from UTC-aligned to Session-anchored Bar Binning

Read this prompt and produce a detailed implementation plan as a markdown file (`PLAN.md`) before writing any code. Do not implement yet.

## Problem

The current Phase 5 partial-bar handling drops any bar whose UTC-aligned bin window is not fully inside `[rth_open, rth_close)`. For RTH (09:30–16:00 ET), this works fine for 1m and 15m (which divide the session evenly) but produces only 6 hourly bars per session (10:00, 11:00, 12:00, 13:00, 14:00, 15:00 ET), silently dropping the 09:30–10:00 opening window.

This is wrong for two reasons:

1. **Analytical:** the opening 30 minutes is the most informative part of the trading day. Dropping it on the 1h timeframe makes 1h regime, volume, and morphology analysis blind to opening dynamics.

2. **Cross-timeframe consistency:** 1m and 15m timeframes have data for 09:30–10:00, but 1h does not. Any correlation, regime mapping, or matrix work that crosses timeframes has an undefined 1h state for the first 30 minutes of every session. That breaks the regime matrix and morphology correlation work downstream.

It also diverges from industry convention. TradingView, Webull, ThinkOrSwim, and similar platforms session-anchor their bins, not UTC-anchor them. Users expect the first bar of an RTH session to start at the open, not at the next UTC top-of-hour after the open.

## Goal

Migrate the bar aggregator to **session-anchored binning** so the first bar of every RTH session always starts at `rth_open` (09:30 ET), and bins proceed forward from there at the configured `bin_ns` interval.

## Design

### Binning rule change

Current: `bin_start_ns = floor(ts_ns / bin_ns) * bin_ns`

New: `bin_start_ns = rth_open_ns + floor((ts_ns - rth_open_ns) / bin_ns) * bin_ns`

This anchors bin grid to the session open instead of the UTC clock.

### Trailing partial handling

When `bin_ns` does not evenly divide the RTH session length (e.g. 1h on a 6.5-hour session), the final bin will be shorter than `bin_ns`. Two reasonable strategies — please evaluate both in the plan and recommend one:

**Strategy A — Emit trailing partial as a shorter bar.**
- Final bar covers `[rth_open + N*bin_ns, rth_close)` and is allowed to be shorter than `bin_ns`.
- Result for 1h on RTH: 7 bars (09:30, 10:30, 11:30, 12:30, 13:30, 14:30, 15:30), with the last covering only 30 minutes.
- Pros: no data loss, matches TradingView behavior, preserves cross-timeframe coverage.
- Cons: bars are not all the same duration; downstream code that assumes uniform bar duration may need adjustment.

**Strategy B — Merge trailing partial into the previous bar.**
- Final bar covers `[rth_open + (N-1)*bin_ns, rth_close)` and is longer than `bin_ns`.
- Result for 1h on RTH: 6 bars (09:30, 10:30, 11:30, 12:30, 13:30, 14:30), with the last covering 90 minutes.
- Pros: bin count is what users see in retail tools that fold partials backward.
- Cons: the merged bar is non-uniform in the other direction (longer than `bin_ns`).

Recommended default: **Strategy A**. It preserves cross-timeframe analytical consistency (every minute of RTH is covered exactly once at every timeframe), and downstream consumers can branch on `bin.duration_ns` if they need to. Strategy B should be available as a config option but not the default.

### Cross-timeframe invariant to preserve

After migration, this invariant must hold for any timeframe `t` and any RTH session:

> The union of bar windows at timeframe `t` covers exactly `[rth_open, rth_close)` with no gaps and no overlaps.

This is the property that makes cross-timeframe analysis sound. The current implementation violates it for 1h. The new implementation must satisfy it for all supported timeframes.
