 In Wyckoff theory, the "Phase" of the market (Accumulation, Reaccumulation, Distribution, etc.) is established on the higher timeframe (HTF), while the "Spring" or "Upthrust" (the actual entry trigger) is executed on the lower timeframe (LTF).

To support this, we need to move from **Independent Contexts** to a **Relational Bias Engine**.

**Phase 6: Directional Bias & Cross-Timeframe Filtering**.

**Goal:** Implement a Directional Bias engine that uses 1h and 15m Regimes to filter and annotate 1m Events and Canonical Entries.

**Architectural Context:**
The system now has 1m, 15m, and 1h bars/regimes pre-computed in DuckDB. Currently, these are siloed. This phase introduces "Alignment"—the ability for an LTF entry to be aware of HTF structure.

**1. Define the "Bias" Logic (New `bias.py` or equivalent):**
- Create a `BiasEngine` that classifies the "Trend State" of a higher timeframe bar based on its Regime (v_rank, d_rank).
- **Bullish Bias:** HTF is in a "Steady/Deep" state (Low-Mid Vol, High Depth) or "Impulsive/Stacked" (Mid-High Vol, High Depth) moving upward.
- **Bearish Bias:** HTF is in a "Slippage/Void" state (High Vol, Low Depth) moving downward or "Heavy/Absorptive" at a top.
- **Neutral/Caution:** HTF is "Climactic" (Rank 5,5) or "Ghost Town" (Rank 1,1).



**2. Cross-Timeframe Data Enrichment:**
- Modify the `api/bars` and `api/events` endpoints. When fetching 1m data, the API should "look up" the concurrent 1h and 15m regime states.
- Every 1m bar object should be enriched with `bias_1h` and `bias_15m` strings (e.g., "BULLISH", "BEARISH", "NEUTRAL").

**3. Event Filtering (The "Wyckoff Filter"):**
- Update the client-side event detection (or canonical entry logic) to support **Bias Filtering**.
- Example: A 1m "Fade" (Reversal) fire should be downgraded or suppressed if the 1h bias is "Strongly Impulsive" in the opposite direction (don't fade a freight train).
- Example: A 1m "Breakout" fire is upgraded to "High Conviction" if it aligns with the 1h and 15m bias (Triple Alignment).

**4. UI Updates:**

### Why this is the "Next Level"
By implementing this, you are coding the **"Composite Operator"** logic from Wyckoff. You are no longer just looking at a 1-minute "Breakout" in a vacuum; you are asking: *"Is the 1-hour 'Big Money' actually supporting this move with depth, or is this a 1-minute liquidity trap?"*

**A Note on Wyckoff "Springs":** In your 1m chart, a "Spring" often looks like an **Absorption Reversal** (High Vol, High Depth at a low). This prompt ensures that the system only tells you to buy that Spring if the 15m/1h context suggests the market is in an "Accumulation" or "Markup" phase.

### How to run a true rebuild end-to-end

The workspace currently has no `.dbn.zst` files locally, so the rebuild step couldn't be executed in-session. When the raw archive is back on disk (typical path: `data/raw/glbx-mdp3-*.trades.dbn.zst`), the rebuild is one command:

```
python -m orderflow_pipeline rebuild \
    --raw-dir data/raw \
    --out-dir data/sessions \
    --db-path data/orderflow.duckdb \
    --symbol ES --session rth --large-print-threshold 100
```

The `rebuild` subcommand drops and recreates the DB, then loops sessions in `1h → 15m → 1m` order so `_stamp_parent_bias` always sees parent rows already written. After it completes, run a smoke check:

```
SELECT timeframe, COUNT(*) AS bars,
       COUNT(vwap) AS w_vwap,
       COUNT(bias_state) AS w_bias,
       COUNT(parent_1h_bias) AS w_p1h,
       COUNT(parent_15m_bias) AS w_p15
FROM bars GROUP BY 1 ORDER BY 1;
```

Expect: `w_vwap == bars` for all tfs; `w_bias == bars` for all tfs (modulo warmup → NEUTRAL but still non-NULL); `w_p1h` = 1m+15m bar count; `w_p15` = 1m bar count.

### Fire-count delta protocol

To document the filter's economic impact run, in two passes against the same session range:

1. Baseline (Phase 6 disabled): launch the dashboard with `?biasFilter=off&showSuppressed=1`. Walk through known trend days. Record `state.canonicalFires.length` per session in a CSV: `date, watchId, fires_off`.
2. Soft mode (default): reload with `?biasFilter=soft`. Record same metric as `fires_soft`. The delta `fires_soft − fires_off` is always `0` because soft mode never drops fires; instead, capture the breakdown by `tag`: count `HIGH_CONVICTION`, `STANDARD`, `LOW_CONVICTION` per session.
3. Hard mode: reload with `?biasFilter=hard`. Record `fires_hard`. The delta `fires_off − fires_hard` is the count of fires the 1h anchor vetoed.

Append the resulting table here under a `### Fire-count deltas` heading once the rebuild has been run. A healthy filter on a known trend day should suppress 30–60% of counter-trend Fade fires while leaving with-trend Breakout fires untouched (HIGH_CONVICTION ratio rises).

### Warmup-gap monitoring at 10:30 ET

Phase 5 regime ranks need 30 bars to populate. On the 1h timeframe that's ~30 sessions of seed history (the seed loader pulls cross-session); on the 1m timeframe it's the first 30 minutes of the session. Together these mean the **earliest a 1m fire can carry a 1h-derived `tag` is 10:30 ET** (RTH open 09:30 + 30×1m + 1h-bin already populated from seed).

When auditing a session:
- Any 1m fire with `bar_time < 09:30 ET` is impossible (RTH gating).
- A 1m fire with `09:30 ≤ bar_time < 10:30 ET` and `tag === null` is **expected** — the 1m regime hadn't warmed up yet and `evaluateBreakoutCanonical / evaluateFadeCanonical` short-circuit `alignment` to `null` during warmup. These should render in the event log with an empty Align cell, no tint.
- A 1m fire with `bar_time ≥ 10:30 ET` and `tag === null` is **a bug** — it means either the LTF bar's `biasH1`/`bias15m` didn't get denormalized, or the seed-history loader returned no parent rows. Spot check by querying:
  ```
  SELECT bar_time, bias_state, parent_1h_bias, parent_15m_bias
  FROM bars
  WHERE timeframe='1m' AND session_date=DATE '<your day>'
    AND bar_time = TIMESTAMP '<your day> 10:35:00';
  ```
  All four columns should be non-NULL. If `parent_1h_bias` is NULL, the half-open join in `_stamp_parent_bias` missed; verify the 1h row for the 10:00–11:00 ET window exists for that session.
