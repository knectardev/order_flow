## 1. Modular Backtesting Architecture

The system should be decoupled into four primary components that interact through an event loop.

| Component | Responsibility | Existing/New |
| :--- | :--- | :--- |
| **Data Engine** | Reads Databento "Trades" schema from **DuckDB**. Streams rows into the event loop one-by-one or in chunks. | **Existing** (Pipeline) |
| **Regime Classifier** | Processes raw trades to compute `v_rank`, `d_rank`, and `bias_state`. Identifies absorption and sweeps. | **Existing** (Pipeline) |
| **Strategy Engine** | Consumes regime data and fires. Evaluates entry/exit criteria based on your strategies (e.g., `Breakout`, `Fade`). | **New Logic** |
| **Simulated Broker** | Tracks virtual capital, manages orders (Market/Limit), applies commissions/slippage, and records P&L. | **New Component** |

## 2. Technical Implementation Plan

### Phase 1: The Execution Simulator (Backend)
Instead of just visualizing events, you need a module that can "own" a position.
* **Order Manager:** Create a Python class in `pipeline/src/orderflow_pipeline/` called `SimulatedBroker`. 
* **State Tracking:** It must track `current_position`, `entry_price`, `unrealized_pnl`, and `realized_pnl`.
* **Accounting:** Use **Mark-to-Market** accounting (especially if trading Futures like ES/MES) where cash is updated at every bar/trade tick to reflect available buying power.

### Phase 2: Strategy "Triggers"
Your requirements already define "Fires" (Breakout, Fade). You need to map these to trade actions:
* **Entry Logic:** `If fire == "★ Breakout" AND h1_bias == "up" → Market Buy`.
* **Exit Logic:** Define a `StopLoss` and `TakeProfit` based on the `vwap` or `value_edge` already present in your data schema.

### Phase 3: Reporting & Visualization (Frontend Integration)
* **DuckDB Storage:** Store backtest results in a new table `backtest_results` with columns: `trade_id`, `entry_time`, `exit_time`, `type`, `pnl`.
* **Dashboard Update:** Add a "Performance" tab to your `orderflow_dashboard.html` that queries a new `/backtest/stats` endpoint to show:
    * **Equity Curve:** Cumulative P&L over time.
    * **Metrics:** Sharpe Ratio, Max Drawdown, and Win Rate.

## 4. Key Performance Metrics to Track

To ensure your regime classification is actually working, your backtester should report performance **per regime**:

| Metric | Trending (High V-Rank) | Ranging (Low V-Rank) |
| :--- | :--- | :--- |
| **Win Rate** | Target higher for Breakouts | Target higher for Fades |
| **Avg. Trade Duration** | Usually longer | Usually shorter |
| **Profit Factor** | Ratio of gross gains to losses | Ratio of gross gains to losses |

### 5. "Backtest Module"
* Account Simulator: A class to track Cash, PositionSize, and RealizedPnL. 
* Trade Logger: A DuckDB table to store trade entries and exits (linked to the fires table).
* Performance Analytics: A Python script (likely in the pipeline/ directory) to calculate the Sharpe Ratio and Drawdowns from the trade log.
* UI Inputs: A new sidebar component in the dashboard to set your $ Capital, Commission/Tick, and Slippage/Ticks.
* Equity Chart: A new canvas strip below the volume band to plot the cumulative P&L.
