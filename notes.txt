The code you provided confirms exactly why you are seeing that "jumping" and "compression" behavior. The issue is not in the sidebar drawing math—**the issue is that the code is intentionally discarding your high-quality API data the moment you start panning.**

### The Culprit: The "Proxy Flash"
In your first snippet (Profile Selection), you have an explicit instruction that forces a drop in data quality during panning:

```javascript
// From your snippet:
if (!isPanned && _lastApiProfile && ...) {
  profile = _lastApiProfile;
} else {
  // THIS IS THE TRIGGER
  // When you pan, the code skips your accurate API profile 
  // and falls back to a low-fidelity proxy calculation.
  profile = computeProfile(profileBars); 
}
```

Because `computeProfile` is a local approximation (often based on 36-bin OHLC data) and your `GET /profile` response is the server-side ground truth, the **POC, VAH, and VAL will never match**. This is why your POC price jumps from $6539.65 to $6460.63 instantly—the renderer is switching from "High Definition" data to "Low Definition" data mid-session.

---

### Recommended Fix: Stop the Proxy Swap

To fix this, you need to change the logic to **prevent the "Proxy Flash."** Instead of falling back to the proxy while panned, you should retain the "last known good" API profile until the new API request resolves.

#### Step 1: Verify the Data Mismatch
Before applying the code change, confirm this is the cause. Add this `console.log` inside the `else` block in your first snippet:

```javascript
} else {
  // Add this:
  console.log("Panning active: Switching to Proxy. API Cached:", !!_lastApiProfile);
  
  profile = computeProfile(profileBars);
}
```
When you pan, if you see "Switching to Proxy" appearing in your console, you have confirmed the swap.

#### Step 2: Implement "Stale" Retention
Modify the `else` block to preserve the `_lastApiProfile` even when `isPanned` is true. This will stop the sidebar from snapping to the wrong POC/width while the new data loads.

Replace your logic with this pattern:

```javascript
// ... existing checks ...

// FIX: Do not force the proxy fallback if we have a valid API profile 
// that we can reuse as a placeholder while the new one fetches.
if (_lastApiProfile) {
  // Use the last known good profile to prevent the "jump"
  profile = _lastApiProfile;
  // (Optional) Mark as "stale" if you want to visually dim the bars
  profile.isStale = true; 
} else {
  // Only fall back to proxy if we have absolutely nothing else
  profile = computeProfile(profileBars);
}

// Ensure the fetch still happens
if (!hasResolvedProfile(fromIso, toIso)) {
  requestProfile(fromIso, toIso, () => Promise.resolve().then(() => drawPriceChart()));
}
```

### Why this fixes the compression
In your second snippet, the compression happens because `visibleMaxBin` (which controls sidebar width) is calculated based on the profile data. 

If you swap the "correct" server-side profile for an "incorrect" proxy profile (which likely has a different `maxBin` or different volume distribution), the math `(v / widthMaxBin)` will immediately produce different widths. By keeping the `_lastApiProfile` (even if it's slightly outdated by one bar), the sidebar geometry will remain stable, and the "razor-thin" or "distorted" bars will vanish. 

You can then add a small visual indicator (like a grey tint or opacity change) when `profile.isStale` is true to let the user know the data is currently updating.
