The good:
The decision to use path_length / range instead of path_length / displacement is the right structural fix. Range is always ≥ displacement, ratio is naturally bounded between 1 and a small reasonable upper number (typically 1-10 for normal bars, higher for very chopy ones), no cap needed, distribution stays well-behaved. This eliminates the entire class of cap-saturation problems rather than just kicking the can.
NULL on range_ticks == 0 is correct — those are degenerate bars where the metric has no meaning, and a NULL is more honest than a synthetic value. The existing NULL-handling in regime classification already handles this gracefully.
Log-space ranking is the right complement. Even with the better-behaved range denominator, the metric is still multiplicative in nature (it's a ratio), so ranking in log-space is more appropriate statistically.
The verification plan (full rebuild + re-diagnostic + SQL tail checks) is concrete and matches what we discussed. Good.
The flag:
This plan does two independent things at once: changes the metric definition AND changes the ranking transformation. If the post-rebuild behavior is different (which it will be), you won't immediately know which change drove what. That's fine if you trust both decisions are individually correct, but it removes your ability to attribute outcomes.
A more cautious sequencing would be: ship the range-denominator change first, rebuild, observe distribution, then decide whether log-space ranking is also needed. With range as the denominator, log-space might not be strictly necessary because the distribution is already well-behaved. Doing both at once is fine if you're confident, but worth being aware of.
The semantic shift worth understanding before approving:
The plan correctly notes this is a "breaking semantic change." Make sure you're internalizing what that means: the new pld_ratio answers a different question than the old one.

Old: "How indirect was the path relative to the bar's net direction?" — captures choppiness vs. trend
New: "How indirect was the path relative to the bar's max extent?" — captures path inefficiency within the bar's range

Concrete example: a bar that opens at 100, runs to 110, comes back to 100, runs to 90, closes at 105. Path length = 35 (10 + 20 + 15 = ... let me redo: up 10, down 20, up 15 = 45). Displacement = 5 (105 - 100). Range = 20 (110 - 90). Old PLD = 45/5 = 9 (very indirect). New PLD = 45/20 = 2.25 (somewhat indirect but bounded by range).
The new metric is less sensitive to bars that close near their open but went on a journey. If your trading intuition says "I care most about bars where price wandered with no net direction," the new metric will mute that signal somewhat. If your intuition says "I care about path inefficiency in absolute terms," the new metric is fine. Worth thinking through which framing matches your actual use case before committing.

Establish the eyeball expectation now so you can verify or be surprised.
