Here's the project plan, scoped to take you from where you are now to a queryable regime-occupancy system with date/time selection. I'll keep it phased so each phase produces something working before moving to the next, and I'll be explicit about what *not* to build at each phase.

---

## Phase 0: Scope and Invariants (before any code)

Decide and write down:

**The aggregator is the boundary.** Everything upstream of the aggregator is raw market data (DBN files). Everything downstream is derived state in DuckDB. The dashboard never touches DBN directly; the aggregator never serves the dashboard directly. This boundary stays clean throughout the project.

**The database is rebuildable.** Any change to aggregation logic (regime classifier, event thresholds) is followed by dropping and rebuilding the database from raw DBN. No in-place migrations of derived data; no special handling for "old" vs "new" rows. This keeps iteration friction low during the period when you're tuning the classifier.

**The dashboard reads, never writes.** The dashboard queries DuckDB but never modifies it. All writes happen through the aggregator. This means no race conditions, no auth, no transaction concerns for the UI layer.

**RTH only, ESM6 only, 1-minute bars only.** Don't generalize prematurely. Single symbol, single session window, single bar size. Generalization happens after the simple version is working.

These four decisions remove a large class of "wait, how should this work?" questions before they come up.

## Phase 1: Database Layer

**Goal.** A working DuckDB database that the aggregator writes to and the dashboard can query, even if both still mostly do their existing thing.

**Schema.** Three tables to start. Don't add tables you don't need yet.

```sql
-- bars: one row per 1-minute bar
CREATE TABLE bars (
  session_date DATE,
  bar_time TIMESTAMP,         -- ts_event of bar start, UTC
  open DOUBLE, high DOUBLE, low DOUBLE, close DOUBLE,
  volume INTEGER,
  delta INTEGER,
  -- derived metrics (filled in Phase 2):
  range_pct DOUBLE,           -- bar range / rolling avg range
  vpt DOUBLE,                 -- volume per distinct price level
  concentration DOUBLE,       -- fraction of volume at top price level
  v_rank INTEGER,             -- 1..5
  d_rank INTEGER,             -- 1..5
  PRIMARY KEY (bar_time)
);
CREATE INDEX idx_bars_session ON bars(session_date);
CREATE INDEX idx_bars_rank ON bars(v_rank, d_rank);

-- events: sparse, one row per detected event
CREATE TABLE events (
  bar_time TIMESTAMP,
  event_type VARCHAR,         -- 'sweep' | 'absorption' | 'stoprun' | 'divergence'
  direction VARCHAR,          -- 'up' | 'down' | NULL
  price DOUBLE,
  PRIMARY KEY (bar_time, event_type, direction)
);

-- fires: one row per canonical entry firing
CREATE TABLE fires (
  bar_time TIMESTAMP,
  watch_id VARCHAR,           -- 'breakout' | 'fade'
  direction VARCHAR,           -- predicted trade direction
  price DOUBLE,
  -- outcome fields, filled later (Phase 5+):
  outcome VARCHAR,             -- 'pending' | 'success' | 'failure' | 'expired'
  outcome_resolved_at TIMESTAMP,
  PRIMARY KEY (bar_time, watch_id)
);
```

Notice what's *not* here: no trades table (raw DBN handles that), no occupancy table (computed on demand from bars), no fires-outcomes audit log (kept simple). Resist adding columns until something needs them.

**Aggregator skeleton.** A single Python script `aggregator.py` that:

1. Reads all DBN files in `data/raw/`.
2. For each session, parses trades into 1-minute bars (existing logic from your current data adapter, ported).
3. Computes derived metrics (placeholder zeros for now — Phase 2 fills them).
4. Writes to DuckDB, replacing existing rows for the same session.

The script is idempotent: running it twice on the same input produces the same database state.

**Local API skeleton.** A FastAPI app `api.py` with three endpoints:

- `GET /bars?from=...&to=...` — return bars in a time range
- `GET /events?from=...&to=...` — return events in a time range
- `GET /fires?from=...&to=...` — return fires in a time range

Plain SQL behind each endpoint. Returns JSON. Maybe 80 lines of Python total for all three.

**Dashboard wiring.** The dashboard's existing data path (which currently consumes bars from your data adapter) gets a parallel option to fetch bars from the API. Keep the existing path working — *do not remove it*. The new path is opt-in via a UI toggle or URL param. This way you can verify the database-fed dashboard renders identically to the in-memory-fed one before depending on the new path.

**Phase 1 done when.** Aggregator runs cleanly on your existing 5 sessions of DBN data. API serves bars for any time range you query. Dashboard, when set to API mode, renders the same chart and matrix as it does in memory mode. No new features yet; just plumbing.

**What you'll resist during Phase 1.** Adding regime computation. Adding the heatmap. Adding date selection UI. The phase is purely about establishing the database layer; the existing in-memory regime placeholder is fine for now.

## Phase 2: Data-Driven Regime Classifier

**Goal.** Replace the synthetic regime computation with one driven by actual bar properties. This is the gap we identified; this phase fixes it.

**Subsystem boundary.** Create a `regime.py` module with one public function: `compute_ranks(bars_df) -> (v_ranks, d_ranks)`. Input is a pandas DataFrame of bars (sorted by time). Output is two equal-length arrays of integer ranks 1-5. Everything else inside the module is implementation detail.

**Volatility input.**
- Per bar: `range = high - low`, `range_normalized = range / EMA(range, span=20)`.
- This is per-bar, range-relative-to-recent. Higher = more volatile relative to recent character.

**Depth input.** Two components, blended:
- VPT: `volume / count_of_distinct_prices_in_bar` (computed at bar-construction time from raw trades, stored on the bar)
- Concentration: `volume_at_modal_price / total_volume` (also computed at bar-construction time)
- Composite: `depth_score = z(vpt) + z(concentration)` where z is z-score normalization within a rolling window

**Ranking layer.**
- Rolling window: last 100 bars (within session only — at session start, fewer than 100 are used).
- For each bar: compute its percentile within the window for both volatility and depth scores.
- Map percentile to rank: [0, 0.2) → 1, [0.2, 0.4) → 2, ..., [0.8, 1.0] → 5.

**Smoothing.**
- Apply a 3-bar simple moving average to both rank values before storing.
- Round to nearest integer at the end.

**Edge cases to handle.**
- First N bars (N=20 or so) have unstable ranks because the rolling window is too small. Output `NULL` rank (and the dashboard shows the matrix as "warming up" for those bars). Explicit, not silent.
- Bars with zero volume (rare but possible during half-days or maintenance breaks). Skip them entirely from rank computation; their rank is NULL.

**Tests.** A small test file with synthetic input that should produce known outputs. E.g., 100 bars all identical → all rank 3. Bars with monotonically increasing range → linearly ascending v_rank. This catches obvious bugs without needing real data.

**Aggregator update.** Modify the aggregator to call `compute_ranks` after computing bars and write `v_rank`, `d_rank`, `range_pct`, `vpt`, `concentration` into the database. Drop and rebuild the database after the aggregator changes.

**Dashboard verification.** Pull up your previously-saved screenshot moments and verify the matrix now reflects real conditions:
- The trending session that produced the false-positive fade fire — does the regime classifier put it somewhere different now?
- The session where the fade fire correctly predicted reversion — does it land in [Active · Normal]?
- A volume-spike sweep bar — does volatility rank jump in the bars around it?

If the answer to all three is "yes, plausibly," the classifier is working. If any answer is "no, looks wrong," debug before continuing.

**Phase 2 done when.** Aggregator produces real V_rank and D_rank for every bar. Dashboard matrix shows data-driven cells that visibly correlate with what's happening on the price chart. Synthetic random walk is fully removed from the regime path. The `evolveSimState` function (or its remnants) is deleted.

**What you'll resist during Phase 2.** Adjusting canonical entry thresholds. Adding new event types. Building the heatmap. The phase is about getting one thing right (the classifier); changes to entries come after you can see how the new classifier behaves.

## Phase 3: Occupancy Aggregation as Queryable State

**Goal.** Build the heatmap of cell occupancy as a queryable view over the database, with selectable time ranges.

**Data layer.** No new tables needed. Cell occupancy is computed on demand:

```sql
SELECT v_rank, d_rank, COUNT(*) as occupancy
FROM bars
WHERE bar_time BETWEEN ? AND ?
  AND v_rank IS NOT NULL
GROUP BY v_rank, d_rank;
```

Add an API endpoint:

`GET /occupancy?from=...&to=...` — returns a 5x5 grid of counts plus total bars in window.

**UI: time range selector.** Above the matrix, add a selector with these options:

- "Current session" — from session start to now
- "Last hour" — rolling 60 minutes
- "Last N sessions" — last 5, 10, 20 (a small dropdown)
- "All loaded" — every bar in the database
- "Custom range..." — opens a date/time picker

The selected range is stored in component state and passed to the occupancy API endpoint.

**UI: heatmap rendering.** The matrix gets a new background layer. Each cell's background opacity is normalized cell occupancy (most-occupied cell at 1.0, empty cells at 0.0). The existing live-state highlight (current cell border, watched cell stripes, fire pulse) overlays on top of the heatmap layer.

A toggle in the matrix header switches between "show occupancy heatmap" and "show probabilistic posterior" (the existing matrix). Both modes show the current cell with the same border treatment.

**Diagnostic enhancement.** Below the matrix, in the existing "most likely / 2nd alternative" status block, add a third stat: "occupancy in selected range." For the current cell, show "this cell occupied X% of selected window." This tells you whether you're in a common or rare regime relative to whatever timeframe you've chosen.

**Phase 3 done when.** You can scrub through a session and watch the live state highlight move across the heatmap background. You can toggle to "all loaded" and see the joint distribution across all 5 sessions. You can pick a custom range and see what the matrix looked like during a specific period. The occupancy API takes <100ms to respond on any range query against your current data volume.

**What you'll resist during Phase 3.** Deciding what the heatmap "means" trade-wise. Adding it as a criterion to canonical entries. Calibrating canonical thresholds against the new occupancy data. Those are Phase 5+ concerns. The phase is about building the visualization; interpretation comes after observation.

## Phase 4: Brushing and Linking

**Goal.** Make every cell in the matrix clickable, filtering the price chart and event log to bars that occupied that cell.

**API addition.** `GET /bars?v_rank=X&d_rank=Y&from=...&to=...` — bars filtered by cell, optionally by range. This is just adding optional WHERE clauses to the existing endpoint.

**UI behavior.**

- Click a matrix cell. The cell gets a "selected" treatment (slightly different border from "current").
- The price chart highlights the bars that occupied that cell within the current range — could be subtle background color, or marker dots above the relevant bars.
- The event log filters to events that occurred on those bars.
- Click the same cell again, or click empty space, to clear the selection.

**Click on canonical fire glyph (★ or ◆).** Same mechanism: filter chart and event log to that fire's bar plus the next N bars (so you can study what happened after the fire). This is where "did this fade actually mean-revert?" becomes a 1-click query.

**Multi-select** (optional, worth deciding now): can the user select multiple cells at once? Probably yes, because "all fade-cell bars across the loaded data" is a useful query. Hold-shift-to-add is the standard pattern.

**Phase 4 done when.** Clicking a cell visibly filters the chart. The interaction feels responsive (under 100ms from click to render). You can use the tool to ask "show me every bar in [Impulsive · Light]" and visually study what those moments looked like across all loaded sessions.

**What you'll resist during Phase 4.** Building outcome tracking. Adding new canonical entries. Calibrating against the new visibility. The phase is about observation power; what you do with that power comes next.
