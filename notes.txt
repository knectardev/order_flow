Gaps worth addressing
1. Phase 2: seek shouldn't become unconditionally async — create seekAsync instead.
The plan says "refactor the inner for loop in seek" but seek has many callers beyond _loadAllSessionsFromApi: the pan handler, jump-to-fire (which restores chartViewEnd + calls _syncCurrentSession), and the slow path inside precomputeAllFires itself. Making seek return a Promise breaks all non-awaiting callers silently. The safer path is a new seekAsync(n, onProgress?) that wraps the batched loop, and only _loadAllSessionsFromApi (and possibly the TF-switch restore seek) get migrated to await seekAsync(...).
2. Phase 2: race condition on TF switch during in-flight async seek.
If the user switches timeframe while an async seekAsync is yielding mid-loop, the next _loadAllSessionsFromApi starts its own seekAsync on a freshly reset allBars. You'd need a cancellation token or generation counter so the stale in-progress seek detects the invalidation and aborts early — otherwise you get state corruption from two concurrent _commitRealBar streams writing into the same singleton. The plan doesn't mention this at all.
3. Fast path: explicitly document the chartViewEnd restore step even with no seeks.
The plan says "restore chartViewEnd if needed (same as today after the two seeks)" but doesn't confirm whether seek() is what clobbers chartViewEnd in the first place. If seek doesn't touch chartViewEnd, the restore in the fast path is a harmless no-op. If it does (e.g. via _commitRealBar → some viewport side-effect), then the fast path bypasses that clobber and the restore is still correct — but this should be verified explicitly rather than assumed, since the requirements doc (§3) is careful about chartViewEnd semantics in panned vs live state.
4. The "slow path still has two seek(N) calls" note is slightly misleading.
When precomputeAllFires is triggered from the pan handler with savedCursor < N, _loadAllSessionsFromApi is not the caller — so there's no prior seek(N) in flight. The slow path is self-contained: seek(N) + copy + seek(savedCursor). Today's bug (the double seek(N)) only manifests when called from _loadAllSessionsFromApi where savedCursor === N. The plan conflates these two contexts slightly in the "slow path" description — worth clarifying so the code comment is unambiguous.
