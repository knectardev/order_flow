Things to resolve before implementation:

The legacy use_regime_filter migration is left ambiguous. The plan offers two options ("map legacy bool to new matrix" OR "keep use_regime_filter for legacy thresholds only and add two new booleans") but doesn't pick one. Pick now. My vote: keep use_regime_filter for legacy thresholds only, add two new booleans. Reason: the legacy flag has been doing two unrelated jobs (rank gate + threshold strictness), and that conflation is exactly what made today's confusion possible. Splitting into use_regime_filter (legacy thresholds), rank_gate_enabled, and trade_context_gate_enabled makes each flag mean one thing. Existing API consumers passing use_regime_filter: true should explicitly opt into the new gates rather than getting them silently — surprising defaults are worse than verbose configs.
The fires-table vs simulate-time gating decision is left as "single place preferred for consistency." Pick the place. Filter at the fires query before _simulate if at all possible — it's the cleaner place because it keeps _simulate agnostic to filter logic, and downstream telemetry (trade counts, entry timestamps) reflects the actual decision points. Filtering inside _simulate would mean fires get loaded then discarded, which makes per-context diagnostics noisier.
signal_trade_context field name (TBD). Name it now. Recommend entry_trade_context for clarity — "signal" is overloaded in the codebase (signal bar, signal type, etc.), whereas "entry" unambiguously points to the bar that produced the trade. Whichever you pick, lock it before anyone else builds tooling against the column.
The "compare all four modes" parallel run is listed as optional. I'd argue it should be in scope from the start because it's the single most useful piece of UI for the diagnostic workflow. Running four sequential backtests manually and pasting numbers into a spreadsheet defeats the point of having an integrated dashboard. But — if it stretches scope significantly, defer it; the MVP works without it.
Null hypothesis path constraint. The plan flags that today's null hypothesis path requires use_regime_filter=true. Decide what the constraint should be under the new model — does null hypothesis require rank_gate_enabled=true, trade_context_gate_enabled=true, either, both, or neither? Probably "either gate enabled" makes sense (you're testing whether some filter beats random), but it's worth being explicit.

One scope addition worth considering:
The diagnostic-before-filtering step is named in the plan but not given any UI affordance. Consider adding a single read-only "per-context breakdown" panel that shows up after any backtest run — a small table with one row per trade_context value showing trade count, win rate, expectancy, P&L contribution. This is essentially a hardcoded version of the SQL query you'd run yourself. Costs maybe a half-day of UI work and pays off every time you run a backtest going forward, because you can see at a glance whether your filter choice is consistent with the per-context data. Without this, every backtest result requires a manual SQL excursion to interpret. With it, the diagnostic loop closes inside the dashboard.
This is the small "in-dashboard per-context histogram" the plan explicitly defers — I'd un-defer it. Even just a table (not a histogram) would be enough. Worth the cost.
